## Contributing to Documentation

### When to Create Documentation

- **UX Reviews**: After evaluating user experience or conducting user research
- **Design Proposals**: Before implementing major UX changes
- **Security Audits**: After reviewing code for security vulnerabilities
- **Technical Design**: Before implementing complex features
- **User Research**: After interviews, surveys, or usability tests

### Documentation Standards

All documentation should include:

1. **TLDR Section** - 3-5 critical bullet points at the top
2. **Executive Summary** - High-level findings and recommendations
3. **Context** - Scope, methodology, and audience
4. **Detailed Content** - Organized by topic or severity
5. **Action Plan** - Prioritized next steps
6. **Metadata** - Date, author, version, dependencies

### File Naming Conventions

- **Living Documents**: Fixed names (e.g., `personas.md`, `jtbd.md`)
- **Date-Stamped Reports**: `{type}-{topic}-{YYYY-MM-DD}.md`
- **Templates**: `{name}-template.md`
